# 数据库 v2（C/S 架构）

客户端不再连接 MySQL。只有 Spring Boot 服务端使用 `booking_app` 账号访问数据库，客户端通过 HTTPS/JSON 调用服务端 API。

## 当前服务器

- 公网 IP：`121.40.96.66`
- 联调 API 地址：`http://121.40.96.66:8080/api`
- 正式环境：备案域名配置完成后使用 `https://<域名>/api`
- MySQL 地址只能由服务端使用 `127.0.0.1:3306`，客户端不得连接公网 IP 的 3306 端口

## 部署顺序

1. 备份旧库：`mysqldump --single-transaction service_booking_system > backup.sql`
2. 新环境执行 `schema-v2.sql`；旧环境先执行它，再执行 `migrate-v1-to-v2.sql`
3. 修改 `create-app-user.sql` 中的密码并执行，服务端用环境变量保存账号密码
4. MySQL 仅监听内网或 `127.0.0.1`，云防火墙不要开放 3306

建议云安全组仅开放 `22`（限制管理来源 IP）、`80` 和 `443`。开发联调期间若临时开放 `8080`，应限制来源 IP，并在 Nginx/HTTPS 上线后关闭。

## 核心改造

- 新增管理员、服务项目、员工可服务项目表
- 预约支持日期、项目、状态、取消与完成时间，并保留历史
- 数据库唯一索引保证同一员工同一天只有一条待服务预约，避免并发重复下单
- 删除业务数据时采用禁用 `is_active`；预约外键使用 `RESTRICT`，避免历史记录被级联清除
- 登录查询只需按 `username` 索引，不建立包含密码的索引

服务端创建预约时仍应使用事务：锁定员工记录，校验账号与项目状态，插入预约；唯一索引冲突应转换成“该员工当天已被预约”的业务错误。
